# PART 2 — CROSS-CUTTING DESIGN PATTERNS

Things that repeat across multiple pages and deserve one fix.

### Pattern 1: Redundancy between a "count" card and a "list that shows the same info"
- Dashboard: Urgent stat + Urgent alert bar
- Property card: "4 vacant" pill + 4 vacant house tiles
- Property header: "$499 spent in 2026" + each card also shows "$X spent in 2026"

**Rule going forward:** if two elements on the same page display the same
underlying number, one must go. Prefer the visual over the text.

### Pattern 2: Dev leftovers in demo data
Three properties/tenants/vendors named "test" / "Test Building" / "Test Plumber"
appear across Properties, Tenants, Vendors, Leases, Reporting. All dev-only.

**Action:** Delete them from `pm_dev_db`. Listed as fix below.

### Pattern 3: Empty pages look broken, not peaceful
Help, Activity log, Inbox (on sparse days), sometimes Announcements — the
empty state is just... nothing. Users assume a bug, not a "you're done" moment.

**Rule:** every page needs a meaningful empty state. Either a helpful
explainer or pre-populated starter content.

### Pattern 4: Sidebar labels are generic
"Dashboard", "Inbox", "Tickets", "Properties"... same as every PM app.
No unique voice.

**Not urgent.** Quietly's voice lives on pages (greeting, All Clear, etc.).
Sidebar being neutral is fine.

### Pattern 5: Browser-rendered dates everywhere are fine, but some server-side ones still leak UTC hour bands
The Crimson "Up late." on the dashboard fires at 1am Eastern (correct) but
if a user is in Vancouver at 10pm, it still says "Up late" because our tz
cache defaults to America/Toronto. Minor — affects nobody since Julia is
in Quebec.

## Bugs found

### 🐛 Bug 1 — Reporting occupancy math is wrong (HIGH)

**Where:** `src/lib/queries.ts` → `getReportingKPIs()`

**What:**
```sql
SELECT COUNT(DISTINCT t.unit_id) FILTER (WHERE t.unit_id IS NOT NULL)::int AS occupied,
       COALESCE(SUM(p.total_units), 0)::int AS total_units,
       ...
FROM properties p LEFT JOIN tenants t ON t.property_id = p.id
```

The `LEFT JOIN` multiplies each property's `total_units` by the number of
tenants at that property. With 5 tenants and a 3-unit building, `SUM`
returns 15 instead of 3.

**Effect:** Reporting page shows "22% occupancy" when the real number
(verified by direct DB query) is **93.75%** (15 of 16 units occupied).

**Fix:** Use a CTE or subquery to SUM total_units once per property:
```sql
WITH totals AS (
  SELECT COALESCE(SUM(total_units),0)::int AS total_units FROM properties
),
occ AS (
  SELECT COUNT(DISTINCT unit_id)::int AS occupied,
         COUNT(DISTINCT id)::int AS tenant_count
  FROM tenants WHERE unit_id IS NOT NULL
)
SELECT * FROM occ, totals;
```

**I will fix this tonight.** Separate commit so you can review.

## Potential runtime issues (not bugs, just fragile paths)

1. **`getAppTimezone()` caches for 60 seconds.** If Julia changes her tz in
   Settings, UI might display the old tz for up to 60s. Acceptable, worth
   documenting.

2. **Media receipt upload directory** (`/app/media/receipts`) is created on
   first upload via `mkdir -p`. If the Docker volume isn't mounted, uploads
   silently succeed but files are lost on container restart. Already mounted
   correctly in compose — just noting the dependency.

3. **Haiku API calls for Smart Summary + Morning Huddle + chat widget** all
   go through `ANTHROPIC_API_KEY`. If Julia's key hits rate limits, all three
   degrade together. Mitigated by caching, but worth monitoring.

4. **n8n reads `tenants.unit_number` text directly** — this is still the
   denormalized cache after C5. Haven't broken n8n, but if anyone ever
   drops `tenants.unit_number` in a future refactor, n8n silently breaks.
   Noted in a comment in `lib/actions.ts` (ensureUnit). Future-self: do
   not remove `tenants.unit_number`.

5. **Audit log `entity_type` enum** was extended in C7 to include `expense`.
   Any row written before Sprint B will have `before_data` missing fields
   that new code might read. The code already handles this safely with
   optional chaining.

## Security notes (honest, not exhaustive)

- Live webhooks `/ticket-management` + `/sms-intake` are still unauthenticated
  (P2 on roadmap).
- `/api/expenses/upload-receipt` accepts any authenticated user's uploads
  with no file-size cap. Could be abused to fill disk. Not critical because
  basic-auth gates the whole dashboard, but worth a `5MB` cap.
- Receipt files served via `/api/expenses/receipt/[filename]` with no auth
  check on the filename itself — any authenticated dashboard user can
  enumerate and read any receipt. Fine for single-landlord, breaks for
  multi-client.
